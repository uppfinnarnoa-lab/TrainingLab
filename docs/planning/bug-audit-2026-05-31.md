## BUGS

### BUG-01 — Dead statistical computation in `updateVO2maxAndPaces`
**File:** `lib/fitness/cache.ts` lines 303–337  
**Severity:** Medium (performance)  
**Status:** Awaiting approval

**Problem:** `updateVO2maxAndPaces` (which runs after every Strava sync) defines `olRaceFilter`,
builds `statActRuns` / `statLapRuns`, then calls `estimateZonesFromStatisticalAnalysis` twice.
The results (`statZonesResult`, `statZonesLapsResult`) are never stored — the `sharedFields`
block that's persisted to the DB intentionally omits them (zone calibration is manual-only).
This runs ~2 seconds of expensive computation on every sync, for nothing.

**Fix:** Remove lines 303–337 from `updateVO2maxAndPaces`. The variables are not referenced elsewhere.
```typescript
// DELETE these blocks entirely from updateVO2maxAndPaces:
const olRaceFilter = ...
const statActRuns = ...
const statLapRuns = ...
const statZonesResult = estimateZonesFromStatisticalAnalysis(...)
const statZonesLapsResult = estimateZonesFromStatisticalAnalysis(...)
```

### BUG-02 — WU/CD activity filter missing from lap-based path in `updateHRZones`
**File:** `lib/fitness/cache.ts` lines 489–509  
**Severity:** Low (minor data contamination)  
**Status:** Not applied — tested in standalone script; WU/CD laps (198 of 4556) are sparse and distributed across slow-pace buckets that already have adequate data, producing identical LT1/LT2/R² estimates with and without the filter. The bug is theoretically correct but has no measurable effect on this dataset.

**Problem:** `wuCdExclude` (which excludes activities named "WU", "CD", "Warm-up", "Cool-down",
"Uppvärmning", "Nedvarvning") is applied to `statRuns` (whole-activity path) but NOT passed to
`buildStatLapRuns`. An activity explicitly named "WU 8km" is excluded from whole-activity
analysis but its laps still flow into `statLapRunsZones`.

In practice this triggers rarely (WU/CD as standalone activities are uncommon), but when it
does, it adds short high-HR or low-HR laps that can bias bucket HR values.

**Fix:** Add `wuCdExclude` as a filter in `buildStatLapRuns`, or pass it as a parameter:
```typescript
const buildStatLapRuns = (olFilter: (a: ActLight) => boolean) =>
  (acts as (ActLight & { laps?: unknown })[])
    .filter(a => /run|trail/i.test(a.sportType) && olFilter(a)
      && wuCdExclude(a.name ?? "")   // ADD THIS
      && Array.isArray(a.laps))
    ...
```

### BUG-03 — `weatherTemp` not fetched → temperature correction silently disabled in production
**File:** `lib/fitness/cache.ts` lines 48–60 (`loadActivitiesLight`)  
**Severity:** Low (reduced accuracy on hot days)  
**Status:** Awaiting approval

**Problem:** `loadActivitiesLight` does not select `weatherTemp`. When activities are mapped to
estimator inputs, no `weatherTemp` is provided, so all laps default to 15°C
(`const temp = r.weatherTemp ?? 15`). The temperature correction
(`temp > 25°C → weight 0.35; temp > 20°C → weight 0.75`) is entirely inactive in production.

The test script includes `weatherTemp` and benefits from the correction; production does not.

**Fix:**
```typescript
// loadActivitiesLight select — add:
weatherTemp: true,

// In buildStatLapRuns mapping, propagate from parent activity:
.map(l => ({
  ...
  weatherTemp: (a as ActLight & { weatherTemp?: number | null }).weatherTemp ?? null,
}))
```

### BUG-04 — `anchorPB` filter inconsistency between auto and manual paths
**File:** `lib/fitness/cache.ts` lines 181 and 581  
**Severity:** Very Low (edge case — only affects users with sub-1500m PBs)  
**Status:** Awaiting approval

**Problem:** In `updateVO2maxAndPaces` anchorPB filters `p.timeSec > 60 && p.distanceM >= 1500`.
In `updateHRZones` no such filter exists. If a user has a stored 400m or 800m PB,
the manual calibration path might select it as the anchor, producing a degraded Riegel prediction.

**Fix:**
```typescript
// updateHRZones — add filter to anchorPB:
const anchorPB = racePBs
  .filter(p => p.timeSec > 60 && p.distanceM >= 1500)   // ADD THIS
  .reduce<RacePB | null>((best, p) => { ... }, null);
```

### BUG-05 — PAV merge uses count-weighted average for `medianHR` but weight-summed `totalWeight` ✅ FIXED 2026-05-31
**File:** `lib/fitness/zones.ts` lines 510–513 (`poolAdjacentViolators`)  
**Severity:** Very Low (theoretical accuracy loss; protects sparse/race-heavy datasets)  
**Status:** Fixed

**Problem:** When PAV merges two adjacent buckets, `medianHR` is computed as a count-weighted
average but `totalWeight` is weight-summed. If one bucket has much higher per-lap weights (e.g.,
race-dominated), the merged HR representative is slightly biased toward the count-heavier bucket
rather than the weight-heavier one. In practice PAV merges are only for near-equal adjacent
buckets so the effect is small.

**Fix:** Use weight-weighted average for `medianHR` in the merge:
```typescript
out.splice(i, 2, {
  pace:        (out[i].pace     * out[i].totalWeight + out[i+1].pace     * out[i+1].totalWeight) / tw,
  medianHR:    (out[i].medianHR * out[i].totalWeight + out[i+1].medianHR * out[i+1].totalWeight) / tw,
  count:       tc,
  totalWeight: tw,
});
```

### BUG-06 — Activity map loads but stalls halfway ✅ FIXED 2026-05-31
**File:** `app/(dashboard)/activities/[id]/activity-map.tsx`  
**Severity:** High (user-visible feature broken)  
**Status:** Fixed

**Root cause:** `useEffect` had no cleanup function. React Strict Mode (enabled in development)
invokes effects twice — once for the simulated mount and once for the real one. Without a
cleanup, the first invocation created a Leaflet map and set `_leaflet_id` on the container.
The `_leaflet_id` guard at the top of the second invocation then returned early, leaving no
valid map in the final rendered state. Additionally, there was no guard checking whether the
component was still mounted after the async `import("leaflet")` resolved.

**Fix applied:**
1. Added `return () => { map?.remove(); map = null; }` cleanup to properly destroy the Leaflet
   instance on unmount — this is the correct React pattern for imperative third-party libraries.
2. Removed the `_leaflet_id` guard entirely — cleanup now handles re-initialization correctly.
3. Added `container.isConnected` check after the async import resolves, so unmount-during-import
   is handled gracefully instead of throwing on a detached element.
4. Increased `invalidateSize` timeout from 50ms to 200ms for safer CSS settling.
5. Captured `ref.current` as `container` before the async boundary so the cleanup closure
   always references the correct DOM element.

**Priority:** Medium

```
-5–0°C     (freezing)
0–5°C      (cold)       ← was "< 5°C"
5–10°C
10–15°C
15–20°C
> 20°C
```

**Implementation notes:**
- Find where temperature buckets are defined and the chart data is computed
- Add the three new buckets at the low end
- Update bucket labels and color scheme (cold end likely blue/cyan tones)
- Minimum count threshold should be same as other buckets (currently appears to be ~10)
- Label display: use `< −10°C`, `−10–−5°C`, `−5–0°C` with proper minus signs (−, not -)

### FC-02 — Larger shade differentiation in activity splits bars
**Component:** Activity splits chart (pace-colored bars in activity detail view)  
**Priority:** Low–Medium

**Request:** The color nuance between bars based on pace should be more pronounced — currently
adjacent bars at similar paces look nearly identical. Increase the contrast range of the
pace-to-color mapping so that moderate pace differences produce visibly distinct shades.

**Implementation notes:**
- Find the pace-to-color mapping function (likely a `colorScale` or similar in the splits chart component)
- Current mapping probably uses a linear scale across the full pace range, which compresses
  all bars toward the middle of the color spectrum when paces are clustered
- Fix: use a tighter reference range (e.g., ±30 sec/km around median lap pace instead of min→max)
  so mid-run pace variation fills the full color range
- Alternative: use a perceptually uniform color scale (oklch or hsl with larger L/C spread)
- Ensure the fastest pace remains clearly distinct from the slowest
