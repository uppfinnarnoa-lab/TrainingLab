## 1. Goals — tre separata buggar i samma feature

Det finns **två parallella goal-system** i appen:
- `AthleteProfile.annualGoals` (JSON, per sport, alltid km) — input i `app/(dashboard)/settings/athlete-profile.tsx` rad 174-196, visas som "{år} — Annual Goal"-widgeten på Dashboard (`app/(dashboard)/dashboard/page.tsx` rad 406-439).
- `Goal`-modellen (sport/metric/period/target, metric kan vara "distance" eller "time") — hanteras i `app/(dashboard)/settings/goals/goals-manager.tsx`, visas som "Training Goals"-widgeten på Dashboard (samma fil, rad ~158-190 för beräkning, rendering tidigare i filen).

### 1a. Tidsmål visas i minuter, ska vara timmar

**Fil:** `app/(dashboard)/settings/goals/goals-manager.tsx`, rad 23 (`metricLabel = { distance: "Distance (km)", time: "Time (min)" }`) och rad 154 (`placeholder={newMetric === "distance" ? "km" : "min"}`).

Målvärdet lagras och anges direkt i minuter (`Goal.target`, unit-agnostiskt fält i schema). Verklig data bekräftar problemet: ett mål satt till "8" (avsett som 8 timmar/vecka) lagras och visas som "8 min/vecka" — ingen rimlig användare sätter ett 8-MINUTERS veckomål. Den faktiska progress-beräkningen (`movingTime / 60` i dashboard/page.tsx rad 171) räknar redan korrekt i minuter internt, så detta är en ren UI/UX-fix, ingen ändring av lagrings- eller jämförelselogiken behövs.

**Fix:** ändra `time`-metrikens INPUT och VISNING till timmar (multiplicera med 60 vid spara, dividera med 60 vid visning), behåll lagring i minuter rakt igenom. Uppdatera `metricLabel.time` → `"Time (hours)"`, `step` för time-inputen (just nu 30 minuter — byt till t.ex. 0.5 timmar), och placeholder-texten. Samma omräkning behövs i `app/(dashboard)/dashboard/page.tsx`s "Training Goals"-widget där `g.target`/`agg.min` visas (sök `min per` i den filen för exakt renderingsplats).

### 1b. "All sports · month" och "All sports · year" visar identiskt värde

**Fil:** `app/(dashboard)/dashboard/page.tsx`, rad 162-164:

```ts
const earliestStart = weekStart < monthStart ? weekStart : monthStart;
const goalActivities = await prisma.activity.findMany({
  where: { userId, startDateLocal: { gte: earliestStart < yearStart ? yearStart : earliestStart } },
  ...
```

**Verifierat rotorsak mot riktig data** (engångsskript, körd och borttagen): `weekStart` (vecko-måndag) är kronologiskt EFTER `monthStart` (månadens 1:a) i nästan alla fall, så `weekStart < monthStart` är FALSKT → `earliestStart = monthStart`. Sedan: är `monthStart < yearStart`? Nej, månadsstart ligger ALLTID efter årsstart (1 jan) → även detta villkor är FALSKT → den slutgiltiga frågan blir `startDateLocal >= monthStart`, **aldrig** `>= yearStart` som avsikten är. Resultat: frågan hämtar bara aktiviteter sedan månadsstart, vilket gör att "year"-bucketen (som SKA omfatta hela året) i praktiken bara innehåller samma data som "month"-bucketen — exakt det observerade symptomet (samma "actual"-tal för båda perioderna).

Bekräftat mot riktig lokal data: en direkt fråga sedan `yearStart` (2026-01-01) gav 172 aktiviteter / 9816 min; sedan `monthStart` (2026-06-01) gav bara 1 aktivitet / 72 min — alltså mycket olika tal, vilket bekräftar att produktionens "1855/1855"-symptom kommer från att frågan aldrig nått tillbaka till årsstart.

**Fix:** hela `earliestStart`-konstruktionen är onödig — `yearStart` är per definition ALLTID den tidigaste av de tre (år ⊇ månad ⊇ vecka), så frågan ska helt enkelt filtrera `startDateLocal: { gte: yearStart }` direkt, ingen ternary-kedja behövs. Ta bort rad 162 och ändra rad 164 till att använda `yearStart` rakt av.

### 1c. "Strength: 0/0 km (NaN%)" — fel mätenhet, inte bara en NaN-bugg

**Fil:** `app/(dashboard)/dashboard/page.tsx`, rad 414-416:

```ts
{Object.entries(goalsThisYear).map(([sport, goalKm]) => {
  const ytdKm = Math.round((ytdBySport[sport] ?? 0) / 1000);
  const pct = Math.min(Math.round((ytdKm / goalKm) * 100), 100);
```

Roten till NaN: `goalKm` saknas/är 0 för "Strength" (ingen km-baserad fångar för den sporten) → `0/0 * 100 = NaN`. Men **användaren har explicit klargjort att detta är fel design, inte bara en presentationsbugg**: Strength (och liknande icke-distansbaserade sporter som Yoga) ska inte ha ett km-mål alls — de ska mätas i TID, precis som `Goal`-modellens `metric: "time"`-alternativ redan stöder.

**Designkrav (bekräftat av användaren):** `AthleteProfile.annualGoals`-systemet måste bli sport-medvetet om mätenhet — distansbaserade sporter (Run, Ride, NordicSki, RollerSki, Swim) fortsätter med km; icke-distansbaserade sporter (Strength, Yoga, Workout) ska istället sätta och visa ett TIMMAR-mål.

**Verifierat:** `SportCategory.isRunningRelated` (schema rad 224) är fel flagga att återanvända här — den är till för att GRUPPERA löprelaterade sporter i en "Running"-bucket vid analys (se `WeekSummaryStrip`), inte för att avgöra "är detta en distanssport". Cykling/skidor är uppenbart distansbaserade men inte `isRunningRelated`.

**Fix (att designa under implementation, inte bestämt i detalj här):**
1. `annualGoals`-strukturen måste utökas från `{ [year]: { [sport]: number } }` (alltid km) till att även bära mätenhet, t.ex. `{ [year]: { [sport]: { metric: "distance" | "time"; target: number } } }`.
2. UI:t i `athlete-profile.tsx` rad 174-196 måste antingen (a) låta användaren välja mätenhet per sport (samma mönster som `goals-manager.tsx`s `metric`-väljare), eller (b) auto-välja baserat på sportnamn med ett enkelt mönster (t.ex. samma regex-familj som redan finns i `lib/planner/colors.ts`s `workoutColor()` för att avgöra "är detta en typisk distanssport" — `/run|trail|virtual|cycl|ride|ski/` osv.) — flagga vilket alternativ som väljs i commit-meddelandet, det är ett UX-beslut inte en given sanning.
3. Dashboard-widgeten (rad 406-439) måste läsa rätt fält beroende på metric, visa "X / Y tim" för tidsbaserade mål istället för "X / Y km", och beräkna `ytdActual` från `movingTime` istället för `distance` för dessa sporter (samma `ytdActivities`-fråga rad 149-156 behöver hämta `movingTime` också, inte bara `distance`).
4. **Skydda mot regression:** lägg till en explicit guard mot `goalKm === 0` (eller mer korrekt, mot division med ett ej-satt mål) oavsett — om en sport saknar mål helt ska den inte renderas i widgeten, inte visa NaN%.

## 2. Strava-resync uppdaterar inte `isRace`/`workoutType` (eller `laps`) på redan synkade aktiviteter

**Bekräftat genom att läsa `lib/strava/sync.ts` i sin helhet.** Om användaren ändrar ett pass' lopp-flagga eller workout-typ PÅ STRAVA efter att det redan synkats, och passet sedan synkas om (daglig cron, manuell "Sync"-knapp, eller webhook), plockar appen INTE upp ändringen — `workoutType`/`isRace` (och i två av tre fall även `laps`) sätts bara vid `create`, aldrig vid `update`:

| Funktion | Fil:rad | `workoutType`/`isRace` i update-blocket? | `laps` i update-blocket? |
|---|---|---|---|
| `syncActivities()` | `sync.ts:105-113` | **Nej** | Nej (inte aktuellt här, ingår inte i create-fältlistan för update) |
| `resyncRecentActivities()` | `sync.ts:185-192` | **Nej** | **Nej** |
| `syncSingleActivity()` | `sync.ts:227-239` | **Nej** | Ja (finns redan) |
| `backfill.ts::runHistoricalBackfill()` | `backfill.ts:130-152` | Ja (finns redan, rad 143-144) | Ja (rad 149) |
| Webhook `update`-event | `app/api/strava/webhook/[secret]/route.ts:104-109` | Anropar `syncSingleActivity()` — ärver dess brist trots att den HAR `laps`, saknar fortfarande `workoutType`/`isRace` | — |

Bara den nattliga historik-backfillen (som de flesta användare bara kör en gång, vid kontoskapande) hanterar detta korrekt. De facto-vägen för "jag ändrade ett pass på Strava och vill att appen ska se det" — webhook, manuell sync, daglig cron — missar alla ändringen.

**Fix:** lägg till `workoutType: data.workoutType, isRace: data.isRace,` i update-blocket för alla tre funktioner (`syncActivities`, `resyncRecentActivities`, `syncSingleActivity`). `syncSingleActivity` har redan `laps` med, lägg bara till de två andra fälten; `resyncRecentActivities` saknar både `laps` och `workoutType`/`isRace` i sitt update-block (rad 185-192) — lägg till alla tre.

### 2b. Korrigerande backfill för senaste 3 månaderna

Efter att fixen i §2 är på plats, kvarstår redan-synkade aktiviteter med en POTENTIELLT inaktuell `workoutType`/`isRace`/`laps` om de blev ändrade på Strava INNAN fixen fanns. Användaren bad explicit om en "kolla senaste 3 månaderna"-körning.

**Plan:** ett engångsskript (eller en admin-trigger-bar endpoint, beslut tas vid implementation beroende på hur ofta detta behövs framöver) som kör `syncSingleActivity(userId, stravaId)` — eller en lättviktig variant av samma logik — för varje aktivitet med `startDate` inom de senaste 90 dagarna, för att tvångsuppdatera `workoutType`/`isRace`/`laps` från Stravas nuvarande data. Respektera Stravas rate limit (samma `BETWEEN_REQ_MS`-mönster som `backfill.ts` redan använder) eftersom detta är N individuella `/activities/{id}`-anrop, en per aktivitet i fönstret.

## 4. Race-prediction-regression — KRITISK, rotorsak bekräftad

**Användarens rapport:** "VDOT verkar bättre nu, men mina race estimat är nästan sämre nu än de var innan förra åtgärden. Maraton och Halvmaraton är galet för dåliga, de snabbare 3k och snabbare är typ lite för långsamma."

### 4a. Rotorsak (bekräftad mot riktig data, inte gissad)

Samma sessions PB-detection-backfill (§-arbetet tidigare denna dag, se `docs/planning/archive/PB_DETECTION_SETTINGS_CONSOLIDATION_PLAN_2026_06_24.md`) skapade — helt korrekt enligt SIN egen, redan validerade regel — två nya `RaceRecord`-rader från lopp-flaggade aktiviteter utan tidigare distans-historik:

| Distans | Tid | Käll-aktivitet | Verklig natur |
|---|---|---|---|
| 15K | 88:26 (5:53/km) | "Åland 2-dagars lång!" | **Orienteringslopp**, hel aktivitet 15,5 km |
| 10 Mile | 102:13 (6:21/km) | "Natt SM!" | **Nattorientering**, hel aktivitet 17,2 km |

Detta är INTE en bugg i PB-detection — `isRace=true` är korrekt satt (Strava `workout_type===1`), och dessa ÄR användarens snabbaste kända resultat på dessa distanser. Problemet uppstår i ETT STEG LÄNGRE BORT: race-time-prediction-koden (`lib/fitness/vo2max.ts`) har en **redan existerande, dokumenterad regel** specifikt byggd för att skydda mot orienterings-snedvridning — men den tillämpas INTE på `racePBs` (RaceRecord-tabellen), bara på `bestEfforts` (Activity-tabellens JSON-fält).

**Bevis, läst direkt i koden:**

`personalizedFatigueExponent()` (`vo2max.ts:261-278`) filtrerar `bestEfforts` till `distance <= 10000` med en explicit kommentar: *"Activity.isRace can't be used to rescue long-distance points here: for this dataset literally every isRace=true activity is an orienteering event... trusting 'isRace' beyond 10K would feed in slow technical-terrain pace as if it were road race data."* Samma fil, samma kommentar, fortsätter: *"RaceRecord PBs (separate, user-confirmed, already validated clean for this runner) remain the only trusted source beyond 10K."*

**Det antagandet var sant när det skrevs** — då innehöll `RaceRecord` bara 15 manuellt inlagda, av användaren själv granskade poster. Det är **falskt nu**: `buildKnownPerformances()` (`vo2max.ts:319-335`) tar emot `racePBs` UTAN distansgräns ("Race PBs — authoritative, always trusted at any distance", rad 312) och matar dem rakt in i den lokala bracket-modellen (`personalizedRacePrediction`) — inklusive de två nya, auto-detekterade, orienterings-färgade posterna. Samma sak gäller `estimateCriticalSpeed()` (`lib/fitness/critical-speed.ts:33-40`): den slår ihop `racePBs` upp till 15 000 m OVILLKORLIGT, och racePBs **vinner över** (skriver över) `bestEfforts` för samma distans — den nya 15K-orienterings-tiden (5:53/km) skriver alltså över vad som annars hade varit en mer pålitlig (om existerande) bestEffort-baserad 15K-punkt i CS-regressionen, och drar hela CS/W'-modellens uppskattade kritiska fart nedåt, vilket sprider sig till BÅDE 15K/HM/Marathon-prediktionerna (CS-röstningen, `criticalSpeedVote()`) OCH den lokala bracket-modellens 15K-ankare (`riegel`-kolumnen, som nu pekar EXAKT på 88:26 — den nya orienterings-tiden — istället för en mer trovärdig siffra).

**Verifierat mot riktig data** (engångsskript, körd och borttagen) med dessa två nya poster aktiva: CS-fit ger `csMetersPerSec=4.178` (3:59/km), `rSquared=0.582`, och de resulterande prediktionerna för 15K (`peak=86:53, riegel=88:26`) och Half Marathon (`peak=128:27, riegel=136:13`) — `riegel`-kolumnen för 15K är bokstavligen identisk med den nya orienterings-tiden, vilket bekräftar att den smutsiga datan direkt styr utdata. (Maraton-PUNKTESTIMATET påverkas inte av detta specifikt — `criticalSpeedVote()` röstar inte bortom 22 000 m — men Maratons RANGE-vidgning (rad 518-522) använder samma förorenade `cs`-objekt, så även den är indirekt påverkad.)

### 4b. Varför "3K och snabbare lite för långsamma" är samma rotorsak, mildare variant

`estimateCriticalSpeed()`s regression (alla punkter ≤15 000 m, `racePBs` övertrumfar `bestEfforts`) påverkas av den FÖRORENADE 15K-punkten även för KORTARE distansers prediktion, eftersom CS/W' är GLOBALA parametrar som `criticalSpeedVote()`s hyperbel-gren (`vo2max.ts:410-412`, gäller alla mål ≤15 000 m, alltså även 3K, 5K, 10K) använder rakt av. En nedåtdragen `csMetersPerSec` gör att ÄVEN kortdistans-röstningen blir något långsammare än den borde — den mildare, men fortfarande verkliga, "lite för långsamma" känslan för 3K och kortare som användaren rapporterar.

### 4c. Fixen (princip, detaljer vid implementation)

**Detta är INTE en bugg i CS-blendningslogiken** (`blendedRacePrediction`, `criticalSpeedVote`, `predictionRange`) — den koden gör exakt vad den ska GIVET pålitlig indata. Fixen hör hemma i HUR `racePBs` filtreras INNAN den når `buildKnownPerformances()`/`estimateCriticalSpeed()`, för att återställa det antagande som höll innan auto-detektering fanns.

**Rekommenderad princip:** tillämpa SAMMA regel på `racePBs` bortom 10K som redan gäller `bestEfforts` — `isManual: true`-poster (användaren har själv granskat och skrivit in dem, exakt det ursprungliga "validated clean"-antagandet) förblir alltid pålitliga vid VILKEN distans som helst; `isManual: false` (auto-detekterade) poster bortom 10K behandlas med samma misstro som `bestEfforts` redan har bortom 10K (exkluderas helt, ELLER vikta ner kraftigt — beslut vid implementation, men exkludering är enklast och matchar `personalizedFatigueExponent`s befintliga mönster rakt av).

**Konkret kodändring att specificera vid implementation:**
- `buildKnownPerformances()` (`vo2max.ts:319-335`) behöver veta `isManual` per `racePB` — signaturen för `racePBs`-parametern måste utökas från `{distanceM, timeSec}` till att även bära `isManual: boolean`, och filtreringen vid rad 329-333 ska skippa `distanceM > 10000 && !isManual`.
- `estimateCriticalSpeed()` (`critical-speed.ts:18, 33-40`) behöver samma utökning av sin `racePBs`-parameter och samma filter innan den slås ihop med `bestEfforts`.
- BÅDA anropsställena som bygger `racePBs`/`RacePB[]` (`lib/fitness/cache.ts::loadRacePBs()` rad 76-81, och motsvarande i stats-sidans cache-miss-fallback om den har en egen) måste `select: { ..., isManual: true }` och skicka det vidare.
- **Uppdatera doc-kommentaren** på rad 312-317 i `vo2max.ts` ("Race PBs — authoritative, always trusted at any distance") — den är nu sakligt FEL och måste skrivas om för att reflektera den nya regeln, annars upprepas samma misstag nästa gång någon läser kommentaren och litar på den.

### 4d. Validering (obligatorisk vid implementation)

1. Återskapa engångsskriptet som körde `computeRacePredictions()`/`estimateCriticalSpeed()` mot riktig data (samma mönster som denna utrednings skript, kört och borttaget) — bekräfta att 15K/HM/Marathon-prediktionerna FÖRBÄTTRAS (rör sig bort från de orienterings-förorenade siffrorna) efter fixen, utan att de KORTA distansernas redan-goda prediktioner (verifierade i förra sessionen mot ±0.5% fel) försämras.
2. Bekräfta specifikt att 15K-radens `riegel`-värde INTE längre är identiskt med den nya auto-detekterade 88:26-posten efter fixen (om det fortfarande är identiskt, är filtreringen inte korrekt kopplad).
3. Detta är INTE en anledning att ångra PB-detection-fixen i sig — `RaceRecord`-datan är fortfarande korrekt FÖR PB-TRACKERN (15K:88:26 ÄR användarens snabbaste 15K, helt korrekt att visa på Races-sidan). Problemet är uteslutande att race-prediction-koden behöver bli mer skeptisk till VILKEN typ av `RaceRecord`-data den litar på vid längre distanser, inte att PB-trackern ska sluta spåra orienterings-resultat.

## 5. Personal Recovery Time — utredd, INGEN bugg hittad

**Fil:** beräkning i `app/(dashboard)/stats/page.tsx` (rad ~829-841), visning i `stats-client.tsx` (rad ~284-296). Formel: hitta varje "trough" där TSB (Training Stress Balance) dyker under −15, mät dagar till TSB återhämtar sig till ≥0, medelvärdet av alla sådana intervall (kräver ≥2 händelser).

**Slutsats:** modellen är fysiologiskt sund (Banister PMC-modellen, 0-gränsen för TSB är standard i sportvetenskap för "återhämtad"). Inga beräkningsfel hittade — ingen hårdkodad placeholder, ingen enhets-förväxling, ingen inverterad jämförelse. Begränsningar som INTE är buggar utan medvetna designval: trösklarna (−15/0) är fasta, inte personaliserade; ingen HRV/sömn-korsvalidering trots att Garmin-data finns tillgänglig för andra delar av samma sida; kräver ≥2 historiska händelser (visar inget om bara 1 finns). Per Bug Audit Practice: dokumenterat varför detta INTE är en bugg snarare än att "fixa" något som fungerar som avsett. Ingen kodändring föreslås för denna punkt.
