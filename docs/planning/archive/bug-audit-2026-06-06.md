# Bug Audit — 2026-06-06

All bugs below were verified by tracing the exact code path. Each is a confirmed functional regression, not a style opinion.

---

## Bug 1: OutcomeModal always closes even when API save fails
**File:** `components/planner/OutcomeModal.tsx:30-35` + `app/(dashboard)/planner/planner-client.tsx:270-283`
**Severity:** High

**Description:** When the user marks a session as completed/missed, the modal closes regardless of whether the PATCH request succeeded. If the server returns an error, the workout state is NOT updated (the `if (res.ok)` guard in `handleOutcomeSave` prevents it), but the modal is already gone and the user sees no indication that the save failed.

**Root cause:**
```tsx
// OutcomeModal
async function save(status: string, r?: string, n?: string) {
  setSaving(true);
  await onSave(workout.id, status, r, n);
  setSaving(false);
  onClose();  // ← always called, even if onSave's fetch returned !res.ok
}

// planner-client
async function handleOutcomeSave(...) {
  const res = await fetch(...)
  if (res.ok) { setWorkouts(...) }  // state update gated on success
  setStatusWorkout(null);           // modal close NOT gated on success
}
```

**Fix:** Gate `onClose()` / `setStatusWorkout(null)` on `res.ok`. Return a boolean from `onSave` or throw on failure, and only close the modal on success.

## Bug 2: OutcomeModal delete fires without await — race condition
**File:** `components/planner/OutcomeModal.tsx:124-128`
**Severity:** Medium

**Description:** The delete confirm button calls `onDelete!(workout.id)` without `await`, then immediately calls `onClose()`. The modal closes before the DELETE request and state update complete. If the user quickly navigates or triggers another action, state and server can be out of sync.

**Root cause:**
```tsx
onClick={async () => {
  setDeleting(true);
  onDelete!(workout.id);  // ← missing await
  onClose();              // ← fires before fetch finishes
}}
```

**Fix:** `await onDelete!(workout.id)` before `onClose()`.

## Bug 3: BlockEditorModal allows end date before start date
**File:** `components/planner/BlockEditorModal.tsx:54,80,197`
**Severity:** Medium

**Description:** The save button only checks `!startDate || !endDate` — it doesn't reject `endDate < startDate`. Creating a block where end is before start results in the block not rendering on any day in the calendar (the `while (cursor <= end)` loop in `blockByDate` never executes). The "X weeks" preview label always shows ≥ 1 due to `Math.max(1, ...)`, masking the invalid input.

**Root cause:**
```tsx
// Line 197 — no date order check
disabled={saving || !name.trim() || !startDate || !endDate}

// Line 79-81 — Math.max hides negative week count
const weeks = Math.max(1, Math.ceil(...));  // always ≥ 1 even when end < start
```

**Fix:** Add `|| startDate > endDate` to the disabled condition and remove `Math.max` so the user sees a negative/zero week count as a warning.

## Bug 4: BlockEditorModal and WorkoutBuilder close unconditionally on save failure
**File:** `components/planner/BlockEditorModal.tsx:67`, `components/planner/WorkoutBuilder.tsx` (parent close logic)
**Severity:** Medium

**Description:** `handleSave` in BlockEditorModal calls `onClose()` after `await onSave(...)`, regardless of whether `onSave` threw or the API returned an error. Same pattern in WorkoutBuilder via `planner-client.tsx handleBuilderSave` which calls `setShowBuilder(false)` before attempting any fetch. If a network error occurs, the builder closes and the user has no way to retry.

**Root cause:**
```tsx
// BlockEditorModal
async function handleSave() {
  setSaving(true);
  await onSave({...});  // if this fails silently, we still close
  setSaving(false);
  onClose();            // ← unconditional
}

// planner-client handleBuilderSave
async function handleBuilderSave(data: BuilderData) {
  setShowBuilder(false);  // ← closes BEFORE any fetch
  // ... then does fetches
}
```

**Fix:** In `handleBuilderSave`, close the builder only after all fetches succeed. In `BlockEditorModal`, only call `onClose()` if `onSave` resolved without error.

## Bug 5: createWorkout and mutation helpers fail silently — no user feedback
**File:** `app/(dashboard)/planner/planner-client.tsx:137-147, 264-267, 329-332`
**Severity:** Medium

**Description:** `createWorkout`, `handleDeleteTemplate`, and `handleDeleteWorkout` all return early on `!res.ok` with no toast, alert, or state change. A network error or server-side validation failure is invisible to the user — the workout never appears, but there's no error message.

**Root cause:**
```tsx
async function createWorkout(data) {
  const res = await fetch("/api/planner/workouts", {...});
  if (!res.ok) return;  // ← silent
  // ...
}
async function handleDeleteTemplate(id) {
  const res = await fetch(`/api/planner/templates/${id}`, { method: "DELETE" });
  if (res.ok) setTemplates(...)  // ← failure = templates list shows deleted item as still present
}
```

**Fix:** Show an error message (even `console.error` + a visible toast) on `!res.ok`. At minimum, `handleDeleteTemplate` should not silently leave the deleted template visible in the list.

## Bug 6: Copy-mode banner says "right-click to paste" but left-click also pastes
**File:** `app/(dashboard)/planner/planner-client.tsx:383-385`
**Severity:** Low

**Description:** The copy-mode banner shows "right-click a day to paste" on desktop, but the day-cell `onClick` handler immediately pastes when a day is tapped in paste mode — no right-click needed. Right-click works too (via context menu), but the primary/simpler action is a regular left-click, which the banner doesn't mention.

**Root cause:**
```tsx
// Banner text
— right-click a day to paste
<span className="md:hidden"> or tap a day</span>

// But PlannerCalendar day onClick:
if (isPasteMode && onPasteWorkout) { onPasteWorkout(key); }  // fires on left-click
```

**Fix:** Change banner text to `"click (or right-click) a day to paste"` on desktop to match actual behaviour.

## Bug 7: WorkoutBuilder date label wrong when editing a planned workout
**File:** `components/planner/WorkoutBuilder.tsx:183`
**Severity:** Low

**Description:** The date field label always reads "Date (leave blank to add to library only)". When the builder is opened via `plannedWorkoutMode` (editing an existing planned workout), leaving the date blank has no "add to library" effect — the workout already exists. The label is confusing.

// Always rendered, even when plannedWorkoutMode=true

## Bug 8: Move mode — clicking a workout pill silently absorbs the tap
**File:** `components/planner/WorkoutPill.tsx:68-71`, `components/planner/PlannerCalendar.tsx`
**Severity:** Low

**Description:** When move mode is active (a workout is being moved), tapping a WorkoutPill on the destination day calls `e.stopPropagation()` inside the pill's `onClick`. This prevents the day cell's `onClick` from firing, so the move never completes. The user also cannot open the tapped workout's detail/edit modal. The only escape is the "Cancel" button in the move banner. This makes move mode feel broken when the destination day already has workouts.

**Root cause:**
```tsx
// WorkoutPill onClick always stopPropagation, preventing day-cell click
onClick={e => {
  e.stopPropagation();
  if (didLongPress.current) { ... return; }
  onClick(workout);  // opens edit/status modal
}}
```

In move mode the user likely wants to drop onto the day, not open the pill's modal.

**Fix:** In `WorkoutPill`, check if a `moveMode` prop is set and skip `stopPropagation` in that case, letting the day cell click handle the move. Or: in move mode, show pills as non-interactive overlays.
