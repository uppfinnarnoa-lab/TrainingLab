## Executive Summary

The codebase is generally well-structured with consistent auth guards and Zod validation on all mutating endpoints. No critical vulnerabilities were found. The most significant issues are: missing rate limiting on AI and sync endpoints (which could exhaust external API budgets), a credentials-cache race condition that could briefly serve one user's OAuth secrets to another, and several logic/null-safety bugs in client components and fitness math.

### 1.3 `approvedAction.toolName` is not validated against the allowed tool set (server-side)
**Severity: HIGH**  
**File:** `app/api/coach/chat/route.ts` lines 18-22, 115

The `approvedAction` field in the chat request body is validated by Zod as `{ toolName: z.string(), toolInput: z.record(z.unknown()) }`. The `toolName` is an arbitrary string that is passed directly to `executeCoachTool()`. While `executeCoachTool()` has a `default:` branch that returns an error for unknown tool names, a crafted `toolName` cannot escape user scoping because all tool operations query by `userId`.

However, `approvedAction.toolInput` is `z.record(z.unknown())` ŌĆö completely unconstrained. The `executeCoachTool` function casts fields like `input.date as string`, `input.workoutId as string`, `input.weightKg as number` without validating types. A malicious payload could pass unexpected types:

```ts
// Attacker sends:
{ toolName: "update_profile", toolInput: { weightKg: "'; DROP TABLE users; --" } }
// Reaches: data.weightKg = input.weightKg  ŌåÆ stored as-is via Prisma (parameterized, so SQL safe)
// But: causes Prisma type-check failure at DB level, silently stored as string in JSON field
```

Prisma parameterizes all queries so SQL injection is not possible, but type coercion issues could corrupt numeric profile fields.

**Fix:** Add per-tool input schemas (re-using Zod) in `executeCoachTool`, or at minimum validate that `toolInput` fields are the correct types before trusting them:

```ts
// In schema:
approvedAction: z.object({
  toolName: z.enum(["create_workout", "delete_workout", "update_profile"]),
  toolInput: z.record(z.unknown()),
}).optional(),
```

### 1.4 `WorkoutType` PATCH/ownership ŌĆö `sportId` cross-user link possible
**Severity: MEDIUM**  
**File:** `app/api/sports/route.ts` lines 53-60

When creating a `WorkoutType`, the server verifies `userId` is attached but does NOT verify that `body.sportId` (a `SportCategory` ID) belongs to the same user. A user could pass a valid `sportId` belonging to another user:

```ts
// Attacker sends: { kind: "type", name: "evil", sportId: "<victim's sportId>" }
// Result: WorkoutType is created under attacker's userId but linked to victim's SportCategory
```

This is a limited-scope IDOR: the attacker cannot read the victim's data through this, but the foreign key `sportId` references another user's row. Cascade deletes on the victim's `SportCategory` would also delete the attacker's `WorkoutType` ŌĆö confusing behavior.

**Fix:** After parsing, verify `sportId` belongs to `session.user.id`:
```ts
const sport = await prisma.sportCategory.findUnique({ where: { id: parsed.data.sportId } });
if (!sport || sport.userId !== session.user.id)
  return NextResponse.json({ error: "invalid_sport" }, { status: 400 });
```

### 1.5 Template `sportId`/`typeId` cross-user link
**Severity: MEDIUM**  
**File:** `app/api/planner/templates/route.ts` lines 51-77, `app/api/planner/templates/[id]/route.ts` lines 37-68

Same pattern as 1.4: `POST /api/planner/templates` accepts `sportId` and `typeId` (both CUIDs) but does not verify they belong to the requesting user. An attacker could link their template to another user's sport/type category. The write only affects the attacker's own row, but creates a dangling or incorrect FK reference.

**Fix:** Before creating the template, verify:
```ts
const sport = await prisma.sportCategory.findUnique({ where: { id: parsed.data.sportId } });
if (!sport || sport.userId !== session.user.id) return 400;
if (parsed.data.typeId) {
  const type = await prisma.workoutType.findUnique({ where: { id: parsed.data.typeId } });
  if (!type || type.userId !== session.user.id) return 400;
}
```

## 2. Input Validation

### 2.1 `from`/`to` query params in `GET /api/planner/workouts` are not validated
**Severity: MEDIUM**  
**File:** `app/api/planner/workouts/route.ts` lines 23-29

The `from` and `to` search params are passed directly to `new Date(...)` without format validation. `new Date("garbage")` produces `Invalid Date`, which Prisma receives as `NaN` and either throws or silently ignores, depending on the adapter.

```ts
if (from) where.date = { gte: new Date(from) };  // no validation
```

An attacker could send arbitrary strings ŌĆö `new Date("1; DROP TABLE")` is `Invalid Date` (NaN), which Prisma will reject with a runtime error (500), leaking a stack trace in development.

**Fix:** Apply the same regex as used in other routes:
```ts
if (from && !/^\d{4}-\d{2}-\d{2}$/.test(from))
  return NextResponse.json({ error: "invalid_date" }, { status: 400 });
```

### 2.2 `batch` query param in backfill routes is not validated
**Severity: LOW**  
**File:** `app/api/strava/backfill-descriptions/route.ts` line 29, `app/api/strava/backfill-splits/route.ts` line 28

```ts
const batchSize = Math.min(50, parseInt(searchParams.get("batch") ?? "30"));
```

`parseInt("abc")` returns `NaN`. `Math.min(50, NaN)` returns `NaN`. `prisma.activity.findMany({ take: NaN })` will likely return all rows (Prisma treats NaN as no limit). The `?stream=true` guard is checked first, but the `batch` param is read in the non-streaming path.

**Fix:**
```ts
const rawBatch = parseInt(searchParams.get("batch") ?? "30");
const batchSize = isNaN(rawBatch) ? 30 : Math.min(50, rawBatch);
```

### 2.3 `workouts/[id]` PATCH: `color` field has no hex validation
**Severity: LOW**  
**File:** `app/api/planner/workouts/[id]/route.ts` line 13

```ts
color: z.string().optional().nullable(),
```

In the POST route (`workouts/route.ts` line 14), color is validated: `z.string().regex(/^#[0-9a-fA-F]{6}$/)`. But the PATCH schema accepts any string for `color`. An attacker could store arbitrary strings in the `color` column, which is then rendered in CSS `backgroundColor` in the planner UI. If the string contains `"javascript:"` or a CSS injection sequence, it could affect rendering.

In practice the field is used in `style={{ backgroundColor: color }}`, which is safe because React does not evaluate CSS injection as script. But storing arbitrary strings is a data quality issue.

**Fix:** Add the hex regex to the PATCH schema as well.

### 2.4 `weather/backfill` `limit` param: integer not validated
**Severity: INFO**

## 5. CSRF

### 5.1 No CSRF token on state-mutating API routes
**Severity: LOW (self-hosted single-user context)**  
**File:** All POST/PATCH/DELETE routes

None of the mutating API routes check `Origin` or use CSRF tokens. They rely on `session` cookies with `sameSite: "lax"` (set in `auth.ts` line 22).

`SameSite=Lax` prevents cross-site form POST submissions from third-party sites (cross-origin navigation with non-simple requests). However, it does NOT protect against same-site subdomains or against attacks where the browser sends a cross-origin GET request that causes a state change (none exist here). For a self-hosted single-user app on a dedicated domain, this risk is negligible.

**Note:** The `strava/callback` and `garmin/callback` GET handlers perform writes (upsert tokens) ŌĆö these are protected because they also require a valid session. Strava/Garmin OAuth state parameter is NOT explicitly validated (see Section 6).

## 6. OAuth Security

### 6.1 Missing OAuth `state` parameter ŌĆö CSRF on OAuth flow
**Severity: HIGH**  
**File:** `lib/strava/client.ts` lines 7-17, `lib/garmin/client.ts` lines 7-16, `app/api/strava/callback/route.ts`, `app/api/garmin/callback/route.ts`

The OAuth authorization URL is constructed without a `state` parameter:

```ts
const params = new URLSearchParams({
  client_id: creds.stravaClientId,
  redirect_uri: redirectUri,
  response_type: "code",
  scope: "read,activity:read_all",
  // ŌåÉ no state parameter
});
```

The callback handler (`app/api/strava/callback/route.ts`) does not verify a state parameter either. This is a textbook OAuth CSRF vulnerability: an attacker could trick the authenticated user into visiting a crafted URL containing an attacker-controlled OAuth `code`, causing the victim's Strava account to be replaced with the attacker's.

In a single-user self-hosted context the risk is lower (the attacker would need to know the callback URL and obtain a valid Strava OAuth code for their account), but this is a real vulnerability for any multi-user extension.

**Fix:**
```ts
// In getStravaAuthUrl: generate and store state
import { randomBytes } from "crypto";
const state = randomBytes(16).toString("hex");
// Store in session/cache keyed to userId with TTL
params.set("state", state);

// In callback handler: verify
const returnedState = req.nextUrl.searchParams.get("state");
const expectedState = await getStateForUser(session.user.id); // from session store
if (!returnedState || returnedState !== expectedState)
  return NextResponse.redirect(new URL("/settings?strava=csrf_error", req.url));
```

### 6.2 OAuth tokens stored in plaintext in `StravaAccount` and `GarminAccount`
**Severity: MEDIUM**  
**File:** `prisma/schema.prisma` lines 62-66, 74-78

Strava and Garmin OAuth access/refresh tokens are stored as plaintext strings in the database. By contrast, Strava/Garmin API credentials (client secrets, AI keys) are encrypted with AES-256-GCM via `encryptIfNeeded()`. This inconsistency means a database dump exposes live OAuth tokens, while a compromise of `AUTH_SECRET` only additionally exposes credential secrets.

For a self-hosted app where the DB is local, the risk is limited. But if an attacker gains read access to the database (e.g., via a backup file), they obtain valid OAuth tokens.

**Fix:** Apply `encryptIfNeeded()`/`safeDecrypt()` to `accessToken` and `refreshToken` in `StravaAccount` and `GarminAccount`, using the same pattern as the AI key/credential storage. Requires a migration.

### 6.3 Redirect URI constructed from `req.nextUrl.origin` ŌĆö open redirect risk
**Severity: LOW (INFO)**  
**File:** `app/api/strava/callback/route.ts` line 17, `app/api/garmin/callback/route.ts` line 13

```ts
const redirectUri = `${req.nextUrl.origin}/api/strava/callback`;
```

`req.nextUrl.origin` is derived from the `Host` header sent by the client. If the app is behind a reverse proxy that doesn't set `X-Forwarded-Host`, or if an attacker can manipulate the `Host` header (e.g., HTTP Host header injection), the redirect URI sent to Strava could be attacker-controlled. Strava validates against the registered "Authorization Callback Domain," which limits exploitation, but it's still worth hardening.

**Fix:** Use a configured `NEXTAUTH_URL` / `APP_ORIGIN` env var instead of deriving origin from the request:
```ts
const origin = process.env.NEXTAUTH_URL ?? `${req.nextUrl.origin}`;
const redirectUri = `${origin}/api/strava/callback`;
```

## 7. Rate Limiting

### 7.1 Login endpoint has no rate limiting
**Severity: MEDIUM**  
**File:** `auth.ts` lines 29-45, `app/(auth)/login/page.tsx`

The NextAuth credentials provider does not apply rate limiting. The `checkRateLimit()` function exists in `lib/rate-limit.ts` and is used on the `/api/settings/password` endpoint, but not on the login flow. An attacker can make unlimited login attempts.

The in-memory rate limiter exists and works. It just isn't applied here.

**Fix:** Add rate limiting in the `authorize` callback:
```ts
async authorize(credentials) {
  const parsed = loginSchema.safeParse(credentials);
  if (!parsed.success) return null;

  // Rate limit by email
  const rl = checkRateLimit(`login:${parsed.data.email}`, 10, 600); // 10 attempts/10min
  if (!rl.allowed) return null; // NextAuth surfaces this as CredentialsSignin error

  const user = await prisma.user.findUnique({ where: { email: parsed.data.email } });
  if (!user) return null;
  const valid = await bcrypt.compare(parsed.data.password, user.passwordHash);
  if (!valid) return null;
  return { id: user.id, email: user.email, name: user.name };
},
```

### 7.2 AI chat endpoint has no request-level rate limiting
**Severity: MEDIUM**  
**File:** `app/api/coach/chat/route.ts`

While there is a budget check (`if (budget > 0 && current >= budget)`), there is no per-request rate limit. A user with a $100 budget could fire 1000 requests in a second, exceeding the budget before the DB update propagates (race condition ŌĆö see 7.3 below).

**Fix:** Add `checkRateLimit(`chat:${userId}`, 10, 60)` ŌĆö 10 requests per 60 seconds ŌĆö before the budget check.

### 7.3 Budget enforcement race condition
**Severity: MEDIUM**  
**File:** `app/api/coach/chat/route.ts` lines 54-63

The budget check reads `current >= budget` and then later increments `currentMonthSpendUsd` at the end of the stream. If two requests are made in parallel before either increments the DB, both will pass the budget check even if together they exceed the budget.

```ts
// Request 1 reads: current=4.90, budget=5.00 ŌåÆ passes
// Request 2 reads: current=4.90, budget=5.00 ŌåÆ passes  (before request 1 writes 5.20)
// Both run, total spend becomes 5.50
```

**Fix:** Use a Prisma `$transaction` with an atomic check-and-increment, or accept the race condition as a known limitation given the single-user context. Document it. An alternative is to use Postgres advisory locks or a `$queryRaw` for an atomic `UPDATE ... WHERE currentMonthSpendUsd + $cost <= budget RETURNING id`.

### 8.4 `credentials` cache singleton could serve wrong user's credentials
**Severity: HIGH**  
**File:** `lib/config.ts` lines 14-41

The credentials cache is a module-level singleton:
```ts
let _cache: { userId: string; creds: AppCredentials; at: number } | null = null;
```

The cache stores a single entry keyed by `userId`. On the first request from User A, the cache is populated. On the next request from User B within the 30-second TTL:

```ts
if (_cache && _cache.userId === userId && Date.now() - _cache.at < CACHE_TTL_MS) {
  return _cache.creds;  // Ō£ō only returned if userId matches
}
```

The check `_cache.userId === userId` correctly prevents User B from getting User A's credentials. However, the *write* path is:

```ts
_cache = { userId, creds, at: Date.now() };
```

This replaces the entire cache with the current user's credentials. In a single-user app this is fine. In a two-user scenario (admin + regular user), if User A's request triggers a cache miss and starts the async DB query, while User B's request also triggers a cache miss and overwrites `_cache` before User A's query resolves, User A will re-execute the DB query next time (not a security issue).

The real risk is the comment on line 26: `// Non-admin users use the shared admin-level API credentials`. This means non-admin users calling `getCredentials()` will read the admin's Strava/Garmin client credentials. This is by design (shared OAuth app). It is fine from a security standpoint as long as these are application-level OAuth credentials (not per-user tokens). Confirmed: the creds are only `clientId`/`clientSecret`, not access tokens.

**Actual severity: INFO** ŌĆö the userId check is present and correct. The comment about non-admin users sharing admin credentials is intended behavior.

## 9. Client-Side Bugs

### 9.1 `useCallback` dependency array in `ChatInterface.tsx` is stale
**Severity: MEDIUM**  
**File:** `components/coach/ChatInterface.tsx` line 162

```ts
const send = useCallback(async () => {
  const text = input.trim();
  if (!text || streaming) return;
  setInput("");
  await sendWithPayload(text);
}, [input, streaming, convId, provider]); // eslint-disable-line react-hooks/exhaustive-deps
```

The `eslint-disable-line` suppresses a warning about missing `sendWithPayload` in the dependency array. `sendWithPayload` closes over `convId`, `streaming`, and `provider` (via the outer function), but it's not itself memoized or included in the dep array. This means `send` captures a stale version of `sendWithPayload` if any of those values change between renders without `input` or `streaming` changing.

In practice, `convId` changes during a conversation (set after first message), and `sendWithPayload` references `convId` at call time via a closure. Since `send` depends on `input` which changes on every keystroke, the stale closure is unlikely to be observed in practice ŌĆö but it's a latent bug.

**Fix:** Either memoize `sendWithPayload` with `useCallback` and add it to the dep array, or remove the `useCallback` on `send` (the function is cheap to recreate and the optimization isn't necessary here).

### 9.3 `processStream` in ChatInterface: incomplete SSE parsing
**Severity: MEDIUM**  
**File:** `components/coach/ChatInterface.tsx` lines 107-154

The SSE stream is parsed by splitting on `"\n\n"`. This works for standard SSE, but:
1. If the server sends a partial chunk that cuts a `data:` line in the middle of a JSON string (e.g., `"data: {"text":"hel"` + `"lo"}\n\n"`), the first split will be `data: {"text":"hel"` which fails `JSON.parse`. The `catch` skips it, but `buffer` is then set to the remainder, which is `"lo"}\n\n"` ŌĆö which also fails to parse. Data is silently dropped.

The `TextDecoder` with `{ stream: true }` should prevent multi-byte character splits, but the `\n\n` split can still split in the middle of a JSON payload if the TCP chunk boundary happens to land there.

In practice, SSE data lines for small payloads like `{"text":"word"}` rarely split, but for larger `toolCall` events it's a real possibility.

**Fix:** Buffer at the line level, not the event level:
```ts
// Standard SSE: a complete event ends with \n\n
// Already correct ŌĆö just ensure the JSON parse failure doesn't swallow events
try { data = JSON.parse(line.slice(6)); } 
catch { console.warn("SSE parse error for:", line.slice(6, 100)); continue; }
```

The current `catch { continue; }` is acceptable, but logging the error helps debugging.

## 10. Logic Bugs

### 10.1 `encryptIfNeeded` "already encrypted" detection is flawed
**Severity: MEDIUM**  
**File:** `lib/encrypt.ts` lines 40-44

```ts
export function encryptIfNeeded(value: string | null | undefined): string | null {
  if (!value) return null;
  if (value.split(":").length === 3) return value; // already encrypted
  return encrypt(value);
}
```

The "already encrypted" heuristic checks for exactly 3 colon-separated segments. An OAuth client secret that happens to contain two colons (e.g., a secret like `abc:def:ghi`) will be incorrectly detected as "already encrypted" and stored as plaintext. Real Strava/Garmin client secrets do not typically contain colons, but this is a fragile assumption.

**Fix:** Use a proper prefix instead of the structure heuristic:
```ts
const ENCRYPTED_PREFIX = "enc:";
if (value.startsWith(ENCRYPTED_PREFIX)) return value;
return ENCRYPTED_PREFIX + encrypt(value);
// Decrypt: strip prefix first
```

Or, more robustly, check that the "encrypted" format is exactly `[32-char hex]:[32-char hex]:[hex]+`:
```ts
if (/^[0-9a-f]{24}:[0-9a-f]{32}:[0-9a-f]+$/.test(value)) return value;
```

### 10.2 `resyncRecentActivities` uses URL string interpolation instead of params
**Severity: LOW / INFO**  
**File:** `lib/strava/sync.ts` line 141

```ts
const listRaw: any[] = await stravaFetch(userId, `/athlete/activities?after=${after}&per_page=30`);
```

`stravaFetch` appends params via `url.searchParams.set()` but this call already has them in the path string. The URL is therefore constructed as `https://www.strava.com/api/v3/athlete/activities?after=...&per_page=30` ŌĆö which works, but is inconsistent with the rest of the codebase where params are passed as the third argument. `new URL(base + path)` will correctly parse the inline query string, so this is functionally correct but fragile.

**Fix:** Pass params as the third argument for consistency:
```ts
const listRaw = await stravaFetch(userId, "/athlete/activities", {
  after: String(after),
  per_page: "30",
});
```

### 10.3 `updateVO2maxAndPaces` does not overwrite `zones` field ŌĆö old zones persist
**Severity: LOW / INFO**  
**File:** `lib/fitness/cache.ts` lines 186-218

`updateVO2maxAndPaces` (the AUTO path) explicitly does NOT update `zones`, `maxHR`, `restHR`, `thresholdHR` in the `update` branch:

```ts
const sharedFields = {
  vo2max, vdot, confidence, method, paces, atl, ctl, tsb, acwr, ...
  // ŌåÉ no zones, no maxHR
};
await prisma.fitnessCache.upsert({
  create: { userId, maxHR, restHR, thresholdHR, zones: existingZones, ...sharedFields },

```

This is intentional (zones are calibrated manually). But `existingZones` in the `create` path is `(existingCache?.zones as object | null) ?? buildHRZonesJson(185, 45)`. **Fix:** Use the computed `maxHR`/`restHR` in the fallback:
```ts
const existingZones = (existingCache?.zones as object | null) ?? buildHRZonesJson(maxHR, restHR);
```

### 10.5 `predictRaceTime` binary search does not guard against VDOT=0
**Severity: LOW**  
**File:** `lib/fitness/vo2max.ts` lines 51-58

```ts
export function predictRaceTime(vdot: number, distanceM: number): number {
  let lo = distanceM / 15, hi = distanceM * 3;
  for (let i = 0; i < 60; i++) {
    const mid = (lo + hi) / 2;
    if (vdotFromRace(distanceM, mid) > vdot) lo = mid; else hi = mid;
  }
  return Math.round((lo + hi) / 2);
}
```

If `vdot` is 0 or negative (which shouldn't happen per the `estimateVO2max` floor of 45 in `lib/fitness/cache.ts`), the binary search will converge on `lo = distanceM * 3` (infinite time), returning an astronomically large number that would render nonsensically in the UI. The caller in `stats/route.ts` does not guard against this.

**Fix:** Add a guard at the call site or within the function:
```ts
if (vdot <= 0) return 0;
```

### 10.6 `tsbAdjustedRaceTime` clamping is asymmetric
**Severity: INFO**  
**File:** `lib/fitness/vo2max.ts` lines 60-63

```ts
export function tsbAdjustedRaceTime(baseTimeSec: number, tsb: number): number {
  const adj = Math.max(Math.min(-tsb * 0.0005, 0.08), -0.04);
  return Math.round(baseTimeSec * (1 + adj));
}
```

The adjustment is clamped to `[-0.04, +0.08]`: up to -4% improvement (very fresh/form) and up to +8% degradation (fatigued). The asymmetry (8% worse vs 4% better) appears deliberate but is worth documenting. With TSB=-160 (extreme overtraining), the result correctly clamps at +8%.

### 11.1 `stravaFetch` does not handle JSON parse errors
**Severity: LOW**  
**File:** `lib/strava/client.ts` lines 69-81

```ts

```

If Strava returns a non-JSON response (e.g., a 502 HTML error page from the CDN), `res.json()` throws a `SyntaxError`. This propagates up as an unhandled exception. ```

### 12.1 `ActivityProfile.yearsTraining` schema is `Int?` but validated as `z.coerce.number()`
