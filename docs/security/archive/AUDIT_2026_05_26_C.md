## Findings

### [HIGH] Security â€” Timing-attack-vulnerable HMAC comparison in OAuth state

**File:** `lib/oauth-state.ts:33`

**Issue:** The HMAC signature comparison uses `sig === expected` (JavaScript string equality). JavaScript string comparison short-circuits on the first mismatched byte, creating a timing side-channel. An attacker who can make many requests and measure latency can brute-force the 16-character hex signature one nibble at a time.

**Risk:** An attacker on the same network (or with repeated measurement) could potentially forge an OAuth state and complete a CSRF attack. For a self-hosted single-user instance this is low-probability but the fix is trivial.

**Fix:**
```ts
import { timingSafeEqual } from "crypto";

// Replace:
return sig === expected;
// With:
return sig.length === expected.length &&
  timingSafeEqual(Buffer.from(sig, "hex"), Buffer.from(expected, "hex"));
```

### [HIGH] Authorization â€” templateId in PlannedWorkout POST has no ownership check

**File:** `app/api/planner/workouts/route.ts:50`

**Issue:** The POST schema accepts `templateId: z.string().cuid().optional().nullable()`. The route creates the workout with the caller-supplied `templateId` directly without checking whether that `WorkoutTemplate` belongs to the authenticated user. A user could supply any valid CUID and link their workout to another user's template.

**Risk:** On a single-user deployment this is a non-issue. However, the schema has an `isAdmin` flag and a comment about "admin-level credentials", suggesting potential future multi-user use. The exploit is read-leakage: when the planner fetches the workout with `include: { template: ... }`, it will return sections from the other user's template. Since the database includes Cascade deletes and the `WorkoutTemplate` itself is protected by separate routes, the actual harm is data leakage of template names and section details.

**Fix:**
```ts
// After parsing, before creating the workout:
if (parsed.data.templateId) {
  const tmpl = await prisma.workoutTemplate.findUnique({ where: { id: parsed.data.templateId } });
  if (!tmpl || tmpl.userId !== session.user.id)
    return NextResponse.json({ error: "not_found" }, { status: 404 });
}
```

### [MEDIUM] Bug â€” Monthly AI spend counter never resets

**File:** `prisma/schema.prisma:330`, `app/api/coach/chat/route.ts:260â€“271`

**Issue:** `AISettings` has a `spendResetAt` field and `currentMonthSpendUsd`/`geminiCurrentMonthSpendUsd` fields, but no code ever reads `spendResetAt` or resets the spend counters. The budget check at `chat/route.ts:70` compares `current >= budget` but `current` only ever increments and never resets to zero. After one month the budget check will always block chat for any user who hit their limit.

**Risk:** Denial-of-service on the AI coach â€” the user can no longer send messages even after the month rolls over. Manual workaround is to update the DB directly.

**Fix:** In `chat/route.ts`, when loading `aiSettings`, check whether `spendResetAt` is more than 30 days ago and reset the counter:
```ts
// After loading aiSettings, before budget check:
const now = new Date();
if (aiSettings?.spendResetAt && (now.getTime() - aiSettings.spendResetAt.getTime()) > 30 * 24 * 3600_000) {
  await prisma.aISettings.update({
    where: { userId },
    data: {
      currentMonthSpendUsd: 0,
      geminiCurrentMonthSpendUsd: 0,
      spendResetAt: now,
    },
  });
  // Re-read or update the local aiSettings object
}
```

### [MEDIUM] Security â€” Strava and Garmin OAuth tokens stored unencrypted

**File:** `prisma/schema.prisma:64â€“65, 76â€“77`

**Issue:** `StravaAccount.accessToken`, `StravaAccount.refreshToken`, `GarminAccount.accessToken`, and `GarminAccount.refreshToken` are stored as plaintext `String` fields. The AI API keys and OAuth client secrets are encrypted, but the actual bearer tokens that grant access to a user's fitness data are not.

**Risk:** Database compromise (SQL injection via Prisma raw queries if any are added, backup file exposure, admin access) directly yields live bearer tokens. For Strava these tokens allow reading all activities and athlete data. This was listed as a HIGH finding in audit B and remains unaddressed.

**Fix:** Encrypt on write and decrypt on read, following the same `encryptIfNeeded`/`safeDecrypt` pattern used for `stravaClientSecret` and `garminClientSecret`. Requires a DB migration to encrypt existing values.

### [MEDIUM] Bug â€” `decrypt()` silently accepts garbage 3-part strings as "legacy plaintext"

**File:** `lib/encrypt.ts:25â€“28`

**Issue:** The `decrypt` function checks `if (parts.length !== 3) return ciphertext`. This is intended as a migration path for old unencrypted values. However, if a value has been corrupted in the DB to contain colons (e.g., `"abc:def:ghi"`) but is not a valid encrypted blob, `decrypt` will attempt AES-256-GCM decryption with 3-byte IV and 3-byte authTag and throw an error â€” which `safeDecrypt` then swallows and returns `null`. The API key becomes unavailable with no log warning, and the route silently falls back to the env-var key or returns `no_api_key`.

**Risk:** Silent data corruption or accidental data loss is not caught early. Low probability but hard to debug.

**Fix:** Add a length check before attempting decryption:
```ts
// In decrypt(), after splitting:
const [ivHex, authTagHex, encryptedHex] = parts;
if (ivHex.length !== 24 || authTagHex.length !== 32) {
  // Matches the ENCRYPTED_RE pattern check â€” return as legacy plaintext
  return ciphertext;
}
```
Since `ENCRYPTED_RE` is already used in `encryptIfNeeded`, you can also reuse it in `decrypt` for early-return consistency.

### [MEDIUM] Security â€” Missing Content-Security-Policy header

**File:** `next.config.ts:6â€“18`

**Issue:** The response headers include HSTS, X-Content-Type-Options, X-Frame-Options, Referrer-Policy, and Permissions-Policy, but there is no `Content-Security-Policy` header. Without a CSP, any stored or reflected XSS would have full script execution capability.

**Risk:** The app renders AI-generated content directly as `whitespace-pre-wrap` text in the chat (not via `dangerouslySetInnerHTML`), and React's JSX escaping prevents straightforward XSS. However, the absence of CSP removes a defence-in-depth layer. If a future change introduces `dangerouslySetInnerHTML` or an `iframe`, there is no policy to contain it.

**Fix:** Add a CSP header to `next.config.ts`. For Next.js with no inline scripts this can be strict:
```ts
{
  key: "Content-Security-Policy",
  value: "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self' https://*.anthropic.com https://*.googleapis.com; frame-ancestors 'none';"
}
```
Note: `unsafe-inline` for styles is common with Tailwind. Adjust `connect-src` to include any CDNs.

### [LOW] Security â€” HMAC state signature is only 16 hex chars (64 bits)

**File:** `lib/oauth-state.ts:16`

**Issue:** `.digest("hex").slice(0, 16)` truncates the HMAC-SHA256 output from 256 bits to 64 bits (16 hex characters). The state token is also constrained to a 10-minute window, which helps. For single-user self-hosted with a long AUTH_SECRET this is not exploitable in practice, but it's below the NIST recommendation of 128-bit MAC tags.

**Risk:** Theoretical brute-force with ~2^64 attempts. In context: negligible.

**Fix:** Remove the `.slice(0, 16)` to use the full 256-bit HMAC, or at minimum use `.slice(0, 32)` for 128 bits.

### [LOW] Bug â€” `resyncRecentActivities` appends query params directly to path string

**File:** `lib/strava/sync.ts:141`

**Issue:** `stravaFetch(userId, "/athlete/activities?after=${after}&per_page=30")` passes query parameters as part of the path string. `stravaFetch` then calls `new URL(STRAVA_BASE + path)` which will correctly parse the embedded query string. However the code style is inconsistent with every other `stravaFetch` call in the file, which uses the `params` argument. If `stravaFetch`'s implementation changes (e.g., normalizing the path), this call breaks silently.

**Risk:** Low â€” correct today, fragile tomorrow.

**Fix:**
```ts
const listRaw = await stravaFetch(userId, "/athlete/activities", {
  after: String(after),
  per_page: "30",
});
```

### [LOW] Bug â€” `planner/blocks` POST/PATCH accepts arbitrary `targetRaceId` without ownership check

**File:** `app/api/planner/blocks/route.ts:15`, `app/api/planner/blocks/[id]/route.ts:15`

**Issue:** `targetRaceId: z.string().optional().nullable()` is stored directly without verifying that the referenced race record belongs to the authenticated user. The foreign key is not enforced in the Prisma schema (`targetRaceId` is a plain `String?`, not a relation).

**Risk:** A user could store another user's race record ID in their training block. Since it's a string reference with no FK, there's no data leakage â€” the value is stored but never joined. The risk is theoretical data poisoning if a future feature reads this field as a relation.

**Fix:** Either add `z.string().cuid()` validation and an ownership check, or document that this is intentionally a loose string reference.

**Risk:** Negligible on single-user deployment. Worth noting for any future multi-user scenario.

### [INFO] In-memory rate limiter resets on process restart

**File:** `lib/rate-limit.ts:4`

**Issue:** The `attempts` Map is in-memory. A server restart (hot reload, OOM restart, deployment) clears all rate limit counters. An attacker who can trigger a process restart (e.g., OOM via large requests) can then immediately attempt 10 more login tries.

**Risk:** On a self-hosted single-user deployment this is acceptable as documented in the code comment. Worth retaining the comment so it's not removed by a future maintainer without consideration.
